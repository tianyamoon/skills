# stock-wiki

## 定位

把 `F:\wiki\trade-system` 当成股票复盘体系的长期真源来维护，不把它当临时笔记堆放区。

这个技能服务 5 类工作：

1. `入库`：把公告、新闻、研报、盘前情报、截图、方法材料等内容放进正确目录。
2. `检索`：按题材、个股、日期、复盘页、方法页、证据链回答问题。
3. `维护`：补页面内容、补双链、补导航、补 `log.md` 留痕。
4. `巡检`：找错路径、孤点页、无效链接、真源漂移、未分发 raw。
5. `路由`：判断“这条内容应该写到哪里”，再决定是否创建或更新页面。

## 先读什么

按任务选择参考文件，不要一次全读：

1. 判断目录落点、真源边界、页面角色时，读 `references/schema-map.md`。
2. 做入库或归档时，读 `references/ingest-rules.md`。
3. 做查询或问答时，读 `references/query-rules.md`。
4. 补题材页、个股页、复盘页、方法页时，读 `references/page-templates.md`。
5. 做结构排查、补链、巡检时，读 `references/lint-rules.md`。
6. 需要判断中文用户会怎么说、如何映射模式时，读 `references/chinese-trigger-examples.md`。
7. 需要直接跑第一轮结构巡检时，读 `references/script-usage.md` 并运行 `scripts/wiki_lint.py`。

## 硬边界

1. `reviews/` 是正式复盘真源。
2. `raw/` 是原始证据真源，不能跳过。
3. `output/` 是交付层，不反向充当事实真源。
4. `review/` 视为旧实验层，不默认写入新正式内容。
5. 页面关系以正文里的 `[[wikilink]]` 为主，不能只靠 frontmatter 的 `sources:`。
6. 涉及历史框架页时，一次只做一个小的、可确认的改动。
7. 事实不确定时显式写 `待核`，不要替用户补脑。

## 工作模式决策

先把用户请求归到下面一种模式，再动手：

### 1. 入库

触发例子：

- “把这篇公告入库”
- “这条盘前消息放进 wiki”
- “把这份研报挂进去”

做法：

1. 先判断材料属于 `announcements / news / intelligence / research / policy / data / assets / methodology` 哪一类。
2. 先落 `raw/`，再决定是否分发到题材页、个股页、复盘页、方法页。
3. 如需正式结论，补到 `reviews/`；如只是证据，不越权写成复盘判断。
4. 重要写入补 `log.md`。

### 2. 检索

触发例子：

- “查一下 PCB 题材最近催化”
- “这只股票在 wiki 里挂了哪些题材”
- “2026-07-05 的复盘页写了什么”

做法：

1. 先查 `reviews/`，再查 `topics/`、`entities/stocks/`、`cycles/`、方法页。
2. 需要追证据时，再回到 `raw/`。
3. 输出时区分 `盘面事实 / 资讯语料 / 复盘判断 / 待核缺口`。

### 3. 维护

触发例子：

- “补一下这个题材页”
- “把这只股票页修完整”
- “把这条线索挂到今天复盘”

做法：

1. 先确认更新的是哪一层页面。
2. 补正文结构、补关联页、补导航或 `log.md`。
3. 不把临时推演硬写成长期结论。

### 4. 巡检

触发例子：

- “巡检一下今天新增页面”
- “看看有没有孤点页”
- “查一下哪里路径写错了”

做法：

1. 优先检查目录错放、失效双链、无入链页面、`review/`/`reviews/` 混用、`output/` 反客为主。
2. 把问题分成：可直接修、建议修、需要确认后再修。
3. 如果只是先做第一轮结构扫描，优先跑 `scripts/wiki_lint.py`。

### 5. 路由

触发例子：

- “这个该放哪里”
- “这算题材还是复盘”
- “这条信息应该挂个股页还是 raw”

做法：

1. 先判断这条内容是证据、知识对象、正式复盘，还是交付产物。
2. 只给一个主落点；如果需要多处体现，用“主落点 + 分发目标”表达。

## 默认输出格式

### 入库类

默认回答：

1. 建议落点
2. 将创建或更新哪些页
3. 关联的题材/个股/复盘/方法页
4. `待核` 项
5. 是否补 `log.md`

### 检索类

默认回答：

1. 结论
2. 证据位置
3. 已有关联页
4. 缺口与待核
5. 如有必要，顺手建议补哪一页

### 巡检类

默认回答：

1. 发现的问题
2. 风险级别
3. 建议修复顺序
4. 可直接修复项
5. 需要用户确认项

## 页面维护原则

1. 页面不是资料堆，要能回答“这页为什么存在”。
2. 题材页要有催化、映射、分支和代表股。
3. 个股页要有主营、题材归属、催化、风险、关联页。
4. 复盘页要能回挂证据与关联节点。
5. 方法页要能指导后续动作，而不只是概念摘录。
6. `index.md` 和 `log.md` 要保持导航真实，不要一边写“待建”一边已经建完。

## 不要做的事

1. 不把 `output/` 内容反向当事实真源。
2. 不把 `01_daily_reviews/` 工作稿直接视为 wiki 正式结论。
3. 不因目录看起来乱，就整仓式重命名历史资产。
4. 不因为用户只问一句，就偷偷扩成大规模重构。
5. 不把“来源提到某股票”自动升级成“该股核心受益”。
